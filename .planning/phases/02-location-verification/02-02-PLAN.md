**Existing patterns:**
- HabitVerificationService.verifyHabit() switches on `habit.type` (not verificationType)
- VerificationType.location already exists in the enum
- AddHabitView.availableVerifications(for:) maps habit types to available verification methods
- AddHabitView already has icon/title/description for `.location` ("location.fill", "Location", "Verify by GPS location")
- Habit uses Codable for UserDefaults persistence — new optional fields must have defaults for backward compatibility

**Geofence verification patterns for 4 habit types:**
1. **Gym / Attend location** → geofence ENTRY at saved location = verified
2. **No fast food** → geofence ENTRY at saved location = FAILURE (reverse logic)
3. **Outdoor time** → time spent OUTSIDE a "home" geofence = verified when target minutes reached

All 4 use CLCircularRegion monitoring with different success/failure logic.

<task type="auto">
  <name>Task 2: Wire location verification into HabitVerificationService</name>
  <files>Core/Services/HabitVerificationService.swift, Core/AppState.swift</files>
  <action>
**HabitVerificationService.swift changes:**

1. Add `private let locationManager: LocationManager` property. Update init to accept it:
   ```
   init(healthKitManager: HealthKitManager, locationManager: LocationManager)
   ```

2. Update `verifyHabit(_:for:)` — BEFORE the existing switch on habit.type, add a guard check:
   ```
   // Location-verified habits use geofence monitoring, not pull-based verification
   if habit.verificationType == .location {
       return verifyByLocation(habit)
   }
   ```
   This way location verification takes priority regardless of habit type.

3. Add `private func verifyByLocation(_ habit: Habit) -> VerificationResult`:
   - If `!habit.hasLocation` → return pending with detail "Set a location to enable verification"
   - If location is set, check if the region is being monitored. If not, start monitoring it.
   - Return `.pending` with detail "Monitoring location..." — geofence verification happens asynchronously via notifications, not pull-based.

4. Add a new method `func handleGeofenceEvent(regionIdentifier: String, entered: Bool)`:
   - This will be called by AppState when it receives geofence notifications
   - Parse habit ID from region identifier (format: "habit-{uuid}")
   - For entry events:
     - For `.noJunkFood` habits → this is a FAILURE (entered forbidden zone)
     - For all other habits (workout, generic attend) → this is a SUCCESS
   - Return a `VerificationResult` so AppState can update todayHabits

**AppState.swift changes:**

1. Update verificationService initialization to pass LocationManager:
   ```
   let verificationService = HabitVerificationService(
       healthKitManager: HealthKitManager.shared,
       locationManager: LocationManager.shared
   )
   ```

2. Add a method `func setupGeofenceMonitoring()`:
   - For each active habit with `.location` verificationType and valid coordinates, call `locationManager.startMonitoring(region:)` with the habit's geofence.
   - Call this in init (after loadHabits) and in addHabit (after adding a location habit).

3. Add geofence notification observers in init:
   ```
   NotificationCenter.default.addObserver(forName: Notification.Name("GeofenceEntry"), ...)
   NotificationCenter.default.addObserver(forName: Notification.Name("GeofenceExit"), ...)
   ```
   When entry notification received: call `verificationService.handleGeofenceEvent(regionIdentifier:entered:true)`, find matching todayHabit, update its status accordingly.

Do NOT implement outdoor time duration tracking yet — that requires a timer mechanism. For now, outdoor time habits with location verification will show "Monitoring location..." as pending. This can be enhanced in a follow-up.
